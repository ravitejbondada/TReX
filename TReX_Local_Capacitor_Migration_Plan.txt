TReX Local Capacitor Migration Plan
===================================

Direction
---------

Move to a 100% local app using Capacitor.

Recommended architecture:

- Capacitor wraps the app as a proper Android/iOS application.
- SQLite is the live local database and source of truth.
- Capacitor Filesystem is used for backup/export/import files.
- Capacitor Share is used to let the user save/share backup files to Drive, iCloud, Desktop, USB, etc.
- Capacitor Local Notifications is used to remind the user to export backups.

Important note:

Do not use a single JSON/cache file as the live database. JSON is fine for backup/export, but SQLite is safer for daily finance data, splits, recurring expenses, filters, sorting, and migrations.


Step 1: Freeze Current Repo
---------------------------

Keep the existing TReX repo as reference only.

Commands:

git status
git add .
git commit -m "Freeze current cloud-enabled TReX app"

Purpose:

- Preserve the current working app.
- Keep all current fixes safe.
- Use it as reference while building the clean local version.


Step 2: Create New Repo
-----------------------

Create a new folder.

Example:

mkdir TReX-Local
cd TReX-Local
git init

Recommended repo name:

TReX-Local

Purpose:

- Clean break from Google/cloud sync.
- No accidental cloud code left behind.
- Easier mental model.


Step 3: Install Required Software
---------------------------------

For Windows/Android development, install:

- Node.js LTS
- Git
- VS Code
- Android Studio
- Java/JDK 21 if Android Studio does not bundle/configure it cleanly

For iPhone/iPad builds:

- Mac required
- Xcode required
- Apple Developer account required for real device/TestFlight/App Store

For now, on Windows, focus on Android first.


Step 4: Create Frontend App
---------------------------

Since the current app is plain HTML/CSS/JS, use Vite vanilla JS.

Command:

npm create vite@latest . -- --template vanilla
npm install

Purpose:

- Gives a clean build system.
- Capacitor can package the built dist folder.
- Keeps the app simple without forcing React/Vue.


Step 5: Copy UI Gradually
-------------------------

From the old repo, copy only useful UI pieces:

- index.html structure
- styles.css
- dashboard/ledger/add expense screens
- category/payment UI
- recurring UI
- credit card UI

Do not copy these directly without review:

- sync.js
- Google auth code
- Google Drive logic
- cloud conflict logic
- remote backup logic
- old localStorage-as-main-store logic

Purpose:

- Keep visual/product work.
- Remove cloud architecture completely.


Step 6: Install Capacitor
-------------------------

Commands:

npm install @capacitor/core
npm install -D @capacitor/cli
npx cap init

Use app values like:

App name: TReX
App id: com.yourname.trex
Web directory: dist

Then build once:

npm run build


Step 7: Add Android Platform
----------------------------

Commands:

npm install @capacitor/android
npx cap add android

After every frontend build:

npm run build
npx cap sync android

To open the native Android project:

npx cap open android

Purpose:

- Creates the real Android wrapper.
- Android Studio handles emulator/device build.


Step 8: Add Official Capacitor Plugins
--------------------------------------

Install official plugins for backup/sharing/notifications.

Commands:

npm install @capacitor/filesystem
npm install @capacitor/share
npm install @capacitor/local-notifications
npx cap sync

Use them like this:

- Filesystem: internal snapshots and backup file creation
- Share: user exports backup to Drive/iCloud/Desktop/USB via OS share/save sheet
- Local Notifications: remind user to export backup daily/weekly


Step 9: Add SQLite
------------------

Capacitor does not have an official SQLite plugin. For a free option, use:

npm install @capacitor-community/sqlite
npx cap sync

Purpose:

- Creates local DB on device.
- Stores real app data safely.
- Avoids JSON-file-as-live-database risk.


Step 10: Design Database Schema
-------------------------------

Start with tables like:

- transactions
- split_items
- categories
- payment_methods
- credit_cards
- recurring_rules
- settings
- app_metadata
- backup_log

Important constraints:

- transactions.id unique
- split_items.transaction_id links to parent transaction
- recurring_rules.id unique
- transactions.recurring_rule_id optional
- one recurring transaction per recurring_rule_id + date

That last constraint prevents duplicate recurring entries at database level.


Step 11: Create Storage Layer
-----------------------------

Do not let screens talk directly to SQLite.

Suggested files:

src/storage/db.js
src/storage/migrations.js
src/storage/transactionsRepo.js
src/storage/categoriesRepo.js
src/storage/recurringRepo.js
src/storage/settingsRepo.js
src/storage/backupRepo.js

The UI should call functions like:

await transactionsRepo.createTransaction(data)
await transactionsRepo.updateTransaction(id, data)
await transactionsRepo.deleteTransaction(id)
await transactionsRepo.listLedger(filters)

Purpose:

- Future changes stay contained.
- Easier testing.
- No messy SQL spread across UI files.


Step 12: Build Migration System
-------------------------------

Create DB versioning from day one.

Example:

version 1: initial tables
version 2: add recurring skipped dates
version 3: add card billing metadata

Store current version in:

app_metadata

Purpose:

- When app updates, old users' databases upgrade safely.


Step 13: Import Existing Data
-----------------------------

Add one-time importer from current backup/local JSON format.

Flow:

1. User opens new app.
2. App detects empty SQLite DB.
3. User chooses old backup JSON.
4. App validates it.
5. App inserts categories, payments, transactions, recurring rules.
6. App marks import complete.

Do not rely on old cloud sync.


Step 14: Rebuild Core Features One By One
-----------------------------------------

Recommended order:

1. Settings and theme
2. Categories/payment methods
3. Add expense
4. Ledger list
5. Edit/delete transaction
6. Split transactions
7. Cumulative sum
8. Dashboard
9. Recurring expenses
10. Credit cards
11. Backup/export/import
12. Notifications

This order is important because ledger and dashboard depend on clean transaction data.


Step 15: Rebuild Cumulative Sum Engine
--------------------------------------

For the new app:

1. Fetch visible ledger rows from SQLite based on current search/filter/sort.
2. Render them.
3. Calculate cumulative amount from bottom to top.
4. Display result per visible tile.

Do not store cumulative sum in DB. It is a display calculation.


Step 16: Rebuild Split Logic Properly
-------------------------------------

Database model:

transactions
- id
- title
- total_amount
- date
- payment_method
- is_split

split_items
- id
- transaction_id
- title
- amount
- category_id

Rules:

- Main amount = sum(split_items.amount).
- Main amount is not editable when split is enabled.
- Delete parent = delete all split items after warning.
- Delete split item = delete only that item.


Step 17: Rebuild Recurring Safely
---------------------------------

Use DB constraints.

Recurring rule:

recurring_rules
- id
- name
- amount
- frequency
- next_due_date
- last_posted_date

Generated transaction:

transactions.recurring_rule_id = recurring_rules.id
transactions.date = due date

Add uniqueness protection:

one transaction per recurring_rule_id + date

This stops duplicate recurring entries permanently.


Step 18: Add Backup System
--------------------------

Use two backup types.

Internal automatic backup:

- App writes local JSON snapshot inside app storage.
- Keep last 7 or last 14 backups.
- Runs when app opens or once per day.

Manual export:

- User taps Export Backup File.
- App creates JSON backup.
- Share plugin opens native share/save dialog.
- User chooses Drive/iCloud/Desktop/USB/etc.

This is OS-compliant and realistic.


Step 19: Add Backup Reminder
----------------------------

Use Local Notifications.

Example:

Every day at 9 PM or user-selected time:
"Backup reminder: export your TReX backup file"

Do not promise silent export to Google Drive/iCloud. The OS generally blocks silent write access to arbitrary personal locations.


Step 20: Test on Web First
--------------------------

Command:

npm run dev

Test:

- add/edit/delete transaction
- split add/delete
- ledger filter/search/sort
- cumulative sum
- recurring generation
- backup export JSON shape
- import restore


Step 21: Test on Android
------------------------

Commands:

npm run build
npx cap sync android
npx cap open android

Then in Android Studio:

- Run on emulator.
- Run on real phone.
- Test app close/reopen.
- Test database persistence.
- Test backup export.
- Test import restore.
- Test notification permission.


Step 22: Remove All Cloud Concepts
----------------------------------

Before first serious release, search for and remove:

- google
- drive
- oauth
- sync
- remote
- cloud
- conflict
- access token
- refresh token

The local app should have no hidden dependency on cloud code.


Step 23: Package Release Build
------------------------------

For Android:

Android Studio -> Generate Signed App Bundle / APK

Later, if publishing:

- Google Play internal testing
- Closed testing
- Production

For personal use, APK install is enough.


Step 24: Keep Docs From Day One
-------------------------------

Create docs in the new repo:

- README.md
- ARCHITECTURE.md
- DATA_MODEL.md
- BACKUP_RESTORE.md
- MIGRATIONS.md
- CHANGELOG.md

Most important one is DATA_MODEL.md. That will keep SQLite sane as the app grows.

Final Recommendation
--------------------

SQLite should be the app's brain.
Capacitor should be the app shell/body.
Filesystem should be the backup locker.

That gives TReX a proper local-first foundation without making the data fragile.
