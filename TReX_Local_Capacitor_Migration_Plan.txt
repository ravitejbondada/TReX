===============================================================================
TReX LOCAL-FIRST MIGRATION PLAN
(Clean Slate Implementation)

STEP 1: ARCHIVE CURRENT CLOUD REPOSITORY

Keep your existing repository completely separate to serve as a read-only
visual reference for your UI layout and CSS styling.

Run these commands in your old project folder:
git status
git add .
git commit -m "Archive cloud-enabled TReX application"

STEP 2: INITIALIZE NEW CLEAN REPOSITORY

Create a brand new folder to ensure absolutely zero legacy Google SDK, sync
scripts, or old localStorage fallback configurations leak into the local code.

Run these commands:
mkdir TReX-Local
cd TReX-Local
git init

STEP 3: BOOTSTRAP & INITIALIZE WORKSPACE WITH VITE (VANILLA JS)

Set up a clean, modern, and lightning-fast asset bundler without taking on the
overhead or structural complexity of a full framework like React or Vue.

Run these commands:
npm create vite@latest . -- --template vanilla
npm install

STEP 4: INSTALL CAPACITOR & CORE MOBILE PLUGINS

Install the core mobile runtime shell alongside the necessary native device
plugins for local file sharing, manual backup handling, and local reminders.

Run these commands:
npm install @capacitor/core @capacitor/filesystem @capacitor/share @capacitor/local-notifications
npm install -D @capacitor/cli

Initialize the project configuration:
npx cap init

(Recommended Prompt Inputs)

App Name: TReX

App ID: com.yourname.trex

Web Directory: dist

STEP 5: INSTALL THE MOBILE SQLITE ENGINE

Install the industry-standard community plugin to establish a secure, native,
structured database directly on the device hardware.

Run these commands:
npm install @capacitor-community/sqlite
npx cap sync

STEP 6: CLEAN-SLATE SQLITE DATABASE SCHEMA (src/storage/migrations.js)

Because you are starting with a blank slate, you can leverage native SQL foreign
key relationships and unique indices immediately to guarantee data integrity.

Create this file to hold your initialization scripts:

export const INITIAL_MIGRATION = `
CREATE TABLE IF NOT EXISTS categories (
id TEXT PRIMARY KEY,
name TEXT NOT NULL,
icon TEXT,
color TEXT
);

CREATE TABLE IF NOT EXISTS payment_methods (
id TEXT PRIMARY KEY,
name TEXT NOT NULL,
type TEXT
);

CREATE TABLE IF NOT EXISTS recurring_rules (
id TEXT PRIMARY KEY,
name TEXT NOT NULL,
amount REAL NOT NULL,
frequency TEXT NOT NULL,
next_due_date TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS settings (
key TEXT PRIMARY KEY,
value TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS transactions (
id TEXT PRIMARY KEY,
title TEXT NOT NULL,
total_amount REAL NOT NULL,
date TEXT NOT NULL,
category_id TEXT,
payment_method_id TEXT,
recurring_rule_id TEXT,
is_split INTEGER DEFAULT 0,
FOREIGN KEY(category_id) REFERENCES categories(id),
FOREIGN KEY(payment_method_id) REFERENCES payment_methods(id),
FOREIGN KEY(recurring_rule_id) REFERENCES recurring_rules(id)
);

CREATE TABLE IF NOT EXISTS split_items (
id TEXT PRIMARY KEY,
transaction_id TEXT NOT NULL,
title TEXT NOT NULL,
amount REAL NOT NULL,
category_id TEXT,
FOREIGN KEY(transaction_id) REFERENCES transactions(id) ON DELETE CASCADE,
FOREIGN KEY(category_id) REFERENCES categories(id)
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_recurring_unique
ON transactions(recurring_rule_id, date);
`;

STEP 7: SUGGESTED REPOSITORY DIRECTORY LAYOUT

Separate your database connection logic from your visual components using the
Repository Pattern. Your user interface will exclusively interact with clean
JavaScript functions rather than writing raw SQL strings inside UI files.

TReX-Local/
+-- src/
|   +-- ui/                     # Your clean UI components and views
|   +-- storage/
|   |   +-- db.js               # Capacitor SQLite connection management
|   |   +-- migrations.js       # DB creation scripts (defined in Step 6)
|   |   +-- transactionsRepo.js # Handles standard CRUD & ledger queries
|   |   +-- settingsRepo.js     # Handles lightweight local app settings
|   +-- main.js                 # Main application loader and entry point
+-- dist/                       # Output folder for production web builds
+-- capacitor.config.json       # Capacitor bridge configurations
+-- package.json                # Dependencies and build scripts

STEP 8: REBUILD AND FUNCTIONAL TESTING EXECUTION ORDER

Because a local database starts completely empty, implement features following
this specific data dependency chain to make testing smooth:

THE SEED LAYER: Build small admin interfaces or setup configurations to
populate your 'categories' and 'payment_methods' tables first.

THE SAMPLE TEST LAYER: Add temporary manual test entries only when needed for
development. Do not build a data reload/import path until you have real data to
preserve.

THE INPUT ENGINE: Port over your "Add Expense" forms to write rows directly
into the native 'transactions' table.

THE LEDGER LIST: Construct your primary history feed utilizing straightforward
SQL select queries with 'ORDER BY date DESC'.

THE SPLIT ENGINE: Implement advanced transaction splits. If a primary parent
transaction row is deleted, the database's native ON DELETE CASCADE rule
will automatically drop child 'split_items' rows without custom code.

THE DISPLAY CALCULATOR: Run cumulative math operations dynamically in memory
on the array subsets returned by your ledger queries. Never save changing
cumulative balances straight back down to the database.

THE MANUAL BACKUP UTILITY: Assemble a clean "Export Backup" feature using
@capacitor/filesystem and @capacitor/share. This converts your active SQLite
state into a structured JSON string and passes it directly to the OS Share
Sheet so users can save backups to external targets (Drive, iCloud, Local).

STEP 9: REQUIRED FEATURE REVAMPS DURING LOCAL MIGRATION

Some current features should not be copied directly. Rebuild these around the
new local SQLite model instead of preserving old sync-era behavior.

TRIP EXPENSES: Remove manual trip-to-ledger sync. Store trip expenses as normal
transactions with an optional trip_id. Show a small trip pill and the actual
trip name in the transaction description or metadata. Ledger, credit-card views,
and trip details should all read from the same transaction row instead of using
duplicate synced records.

GOALS: Rebuild goals with explicit active, completed, and archived states.
Completed goals should leave the active list and move into a completed history
section with final amount, target amount, completed date, and a reopen option.
Goal creation and editing should use focused modals instead of always-open
forms.

TRIPS: Rebuild trips with upcoming, active, completed/past, and archived states.
Past trips should stay available as summary records showing budget, actual
spend, dates, and linked transactions. Trip creation and editing should use
focused modals instead of persistent forms.

TRIP EDIT / DELETE RULES: Allow edit/delete wherever the transaction is visible
as long as the trip pill is clearly shown. The Trips section remains the focused
trip view, but it should not be the only place where a trip-linked transaction
can be managed.

RECURRING EXPENSES: Rebuild recurring as a schedule engine that creates normal
transaction rows when due. Keep schedule editing separate from transaction
editing. Deleted/edited generated dates should be tracked so the engine does not
recreate them accidentally.

EMI / FORECLOSURE: Rebuild EMI schedules as first-class local records with clear
posting history, outstanding principal calculation, and centered modal flows for
schedule, prepayment, and foreclosure actions.

PRESETS: Keep presets as form-fill shortcuts only. Do not create one-tap instant
logging sections in ledger or dashboard. Presets should stay compact near the
top of Add Expense and must never push the save action below the bottom nav.

CREDIT CARD TRANSACTIONS: Treat card spends as normal transactions linked to a
payment method/card. Card views should filter and summarize the same transaction
data instead of maintaining separate expense copies.

CLOUD / SYNC CODE: Do not migrate Google Drive sync, OAuth, conflict modals,
reset-boundary logic, or reload/apply-remote flows into the local repo. Replace
them with explicit local export/import backup utilities only when real data
needs preservation.

STEP 10: ACTIVATE MOBILE BUILDS

Once your code runs reliably inside your web browser environment, activate and
compile your real native Android application shell.

Run these commands:
npm run build
npm install @capacitor/android
npx cap add android
npx cap sync android

To compile the real app and open it inside an emulator or on a physical test
phone via Android Studio, execute:
npx cap open android
